# Decision Log — Capture Forge Product Brief

## 2026-06-19

### D-001: Product Vision Hierarchy
**Decision:** AudienceLens + Semantic Recorder est la vision directrice, mais pas la promesse principale de V0.1. Le positionnement de lancement est « recorder souverain (local-first, privacy-first, modulaire) ». La vision sémantique est révélée en V1+.
**Rationale:** Le marché comprend déjà un recorder privé sans verrouillage. « Capturer l'intention » est trop conceptuel pour être le message principal avant preuve d'exécution. Screenity valide le positionnement « privacy-friendly, open source, no sign-in ».
**Source:** Conversation Herold, 2026-06-19.

### D-002: Licence
**Decision:** MIT pour le cœur original, sous réserve de compatibilité avec le code hérité (Screenity/capture-forge). GPLv3 uniquement si contrainte d'héritage.
**Rationale:** MIT maximise l'adoption, les contributions externes, et la réutilisation dans un écosystème plugins/modules. Éviter la zone grise « on part de Screenity puis on verra ».
**Source:** Conversation Herold, 2026-06-19.

### D-003: Distribution V0.1
**Decision:** GitHub + Chrome Web Store. GitHub pour alpha/nightly/discussions techniques ; CWS pour distribution officielle et crédibilité utilisateur.
**Rationale:** CWS seul = pas d'itération rapide. GitHub seul = pas d'apprentissage du marché réel.
**Source:** Conversation Herold, 2026-06-19.

### D-004: Support Firefox
**Decision:** P1 (après V0.5), pas P0.
**Rationale:** Le manque d'équivalent à tabCapture + offscreen dans Firefox rend le coût de parité trop élevé pour V0.1. Préparer une abstraction capture backend dès maintenant, mais ne pas promettre la symétrie au lancement.
**Source:** Conversation Herold, 2026-06-19.

### D-005: Relecture et corrections brief (after review)
**Decision:** Appliquer 5 corrections au brief suite à la relecture : (1) reformuler Screenity sans mention de paywall, (2) remplacer « le seul » par « combine », (3) ajouter ICP prioritaire V0.1 et aligner personas avec scope WebM, (4) remplacer métriques vanity par métriques fiabilité/performance, (5) ajouter sections Product Principles et Risques Produit.
**Rationale:** Le brief était bon dans sa direction mais présentait des fragilités : affirmations sur les concurrents difficiles à vérifier dans le temps, claims « uniques » impossibles à défendre durablement, décalage entre promesses personas et scope réel, métriques trop orientées vanity.
**Source:** Relecture Herold, 2026-06-19.

### D-006: Section Problem sans référence à des concurrents spécifiques
**Decision:** Remplacer la liste détaillée de concurrents (Loom, Screenity, Awesome Screenshot, OBS) par une formulation générique sur les frictions du marché.
**Rationale:** Les prix, limites et plans changent vite. La liste devenait un passif de maintenance et des affirmations risquaient d'être contestables quelques mois plus tard.
**Source:** Relecture Herold, 2026-06-19.

### D-007: Scope V0.1 minimaliste
**Decision:** Retirer le fallback IndexedDB du scope V0.1 visible dans le brief (gardé techniquement si nécessaire). Simplifier la mention des feature flags dans le message produit.
**Rationale:** Herold recommande un scope brutalement minimal — ne pas sur-exposer les mécanismes de résilience technique dans le message produit tant que le socle OPFS n'est pas validé.
**Source:** Relecture Herold, 2026-06-19.
