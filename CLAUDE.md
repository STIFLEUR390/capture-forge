## Project Overview

**Capture Forge** — a browser extension (Chrome + Firefox planned) for screen recording and video editing, written in Rust and compiled to WebAssembly via the [oxichrome](https://crates.io/crates/oxichrome) framework (wraps `wasm-bindgen` for Chrome extension background service workers in Rust).

Three phased sub-products (per `docs/prd.md`):
1. **Recorder Core** (P0) — screen/tab capture + microphone + pause/resume + stop + WebM export + OPFS storage + crash recovery
2. **Editor** (P1) — timeline-based video trimming and composition
3. **AI/Enrichment** (P2) — post-capture enhancement

**Current state:** Scaffold (MVP not yet implemented). The extension registers a service worker that logs startup and install events.

## Prerequisites

```bash
# Required toolchain
rustup target add wasm32-unknown-unknown
cargo install wasm-pack
# Optional: only needed when regenerating manifest/JS shims
cargo install cargo-oxichrome
```

## Build Commands

```bash
# Standard workflow — recompiles Rust → WASM (fast)
wasm-pack build --target web

# Full oxichrome pipeline — regenerates manifest.json, background.js, and JS shims
cargo oxichrome build
cargo oxichrome build --release  # Optimized build
cargo oxichrome clean            # Remove dist/

# Fast compile-time validation (no wasm target needed)
cargo check
```

The checked-in `dist/chromium/manifest.json` and `dist/chromium/background.js` were generated by oxichrome once. Edit them only when permissions or entry points change.

### Build output

```
dist/chromium/
  manifest.json           # Manifest V3 (hand-edited when adding permissions/entry points)
  background.js           # ES module service worker — imports and calls WASM exports
  wasm/
    capture_forge.js      # wasm-bindgen JS glue (regenerated on every wasm-pack build)
    capture_forge_bg.wasm # Compiled WebAssembly binary
    capture_forge.d.ts    # TypeScript declarations
```

## Development Workflow

1. Edit `src/lib.rs` — all Rust extension logic lives here
2. Run `wasm-pack build --target web`
3. Load `dist/chromium/` as an unpacked extension at `chrome://extensions/` (Developer mode on)
4. Inspect the service worker via Extensions → Capture Forge → Service Worker → Console

If you change permissions, extension name, or version, update both `src/lib.rs` **and** `dist/chromium/manifest.json`.

### Adding a popup or options page

Add `#[oxichrome::popup]` or `#[oxichrome::options_page]` to a Leptos component function in `src/lib.rs`, then run `cargo oxichrome build` (not `wasm-pack` alone — the CLI generates the HTML/JS shims). **Note:** Leptos is not yet in `Cargo.toml` — add it when you implement a popup.

## Architecture

### Crate layout

```

docs/              # Product docs: PRD, product brief, UX specs, architecture, brainstorming
_bmad/             # BMAD methodology workspace (config, agent personas, planning scaffolds)
_bmad-output/      # BMAD-generated artifacts (planning, implementation, tests)
```

### Extension lifecycle

The single Rust entry point uses three oxichrome proc-macro attributes:

1. **`#[oxichrome::extension(...)]`** — declares the extension struct (name, version, permissions). Generates manifest.json metadata and the JS↔WASM binding layer.
2. **`#[oxichrome::background]`** — async function that runs when the service worker starts.
3. **`#[oxichrome::on(runtime::on_installed)]`** — handler for `chrome.runtime.onInstalled`.

### Data flow

```
src/lib.rs (Rust)
    │ oxichrome proc macros + wasm-bindgen
    ▼
dist/chromium/wasm/capture_forge.js + capture_forge_bg.wasm
    │ background.js imports and calls __oxichrome_bg_start()
    ▼
Chrome service worker (Manifest V3)
```
